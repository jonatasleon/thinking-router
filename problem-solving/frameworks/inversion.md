# Framework Name

Inversion

## Core Mental Model

A problem becomes clearer when you reverse it and ask how to cause failure, then avoid those conditions.

## When To Use

- The user is stuck on ideal solutions only.
- The task is high stakes and avoiding mistakes matters.
- You want to expose hidden failure modes.

## When NOT To Use

- The task needs constructive ideation first.
- The problem is already defined by explicit failure criteria.
- The user is asking mainly for relationship mapping.

## Key Diagnostic Questions

- If we wanted this to fail, what would we do?
- What conditions would almost guarantee a bad outcome?
- Which of those conditions already exist?
- How can we avoid or neutralize them?

## Step-by-Step Reasoning Process

1. State the desired outcome.
2. Reverse it into the failure outcome.
3. List actions, conditions, and assumptions that would create failure.
4. Highlight which failure drivers are realistic risks.
5. Convert each major failure driver into a prevention rule or safeguard.
6. Return the risk list and the prevention plan.

## Example Application

If the goal is a successful launch, ask what would guarantee a failed launch: unclear messaging, missing QA, no owner, weak support readiness. Then design safeguards against those conditions.
