# Framework Name

Abstraction Laddering

## Core Mental Model

Stuck problems often improve when you move up to the broader purpose or down to the concrete mechanism. Better framing creates better solutions.

## When To Use

- The user may be solving the wrong problem.
- A problem statement feels too narrow or too vague.
- You need new framing before ideation.

## When NOT To Use

- The problem is already well defined and needs execution.
- Root-cause diagnosis is the main task.
- The issue is straightforward prioritization.

## Key Diagnostic Questions

- What larger goal does this problem serve?
- What more concrete subproblem sits beneath it?
- What changes if we move one level up?
- What changes if we move one level down?

## Step-by-Step Reasoning Process

1. Write the current problem statement.
2. Ask "why does this matter?" to move up the ladder.
3. Ask "how does this happen?" to move down the ladder.
4. Generate alternative framings at higher and lower levels.
5. Compare which framing makes solution paths clearer.
6. Choose the framing with the most leverage and least distortion.
7. Return the original framing, alternative framings, and preferred one.

## Example Application

"Users are not clicking this button" may ladder up to "users are not seeing value" or down to "the call to action is not visible," leading to very different solutions.
