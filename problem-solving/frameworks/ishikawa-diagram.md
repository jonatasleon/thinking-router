# Framework Name

Ishikawa Diagram

## Core Mental Model

Problems usually arise from multiple categories of causes. Root-cause analysis improves when causes are grouped rather than brainstormed as a flat list.

## When To Use

- The user asks for root causes.
- A problem may have multiple contributing factors.
- You need to structure causal hypotheses before acting.

## When NOT To Use

- The task is generating solutions rather than diagnosing causes.
- The problem is actually a decision between options.
- There is no observable problem statement yet.

## Key Diagnostic Questions

- What is the exact effect or problem?
- What major cause categories should be considered?
- What specific causes sit under each category?
- Which causes are most plausible or evidenced?

## Step-by-Step Reasoning Process

1. State the effect or problem at the head of the diagram.
2. Choose the major causal categories appropriate to the context.
3. Brainstorm candidate causes under each category.
4. Push each cause deeper by asking why it happens.
5. Separate evidence-backed causes from speculation.
6. Identify the few high-probability root causes worth testing.
7. Return the cause tree and next investigation step.

## Example Application

If delivery dates keep slipping, separate causes into planning, staffing, scope, dependencies, and process, then identify which branches explain the slippage best.
